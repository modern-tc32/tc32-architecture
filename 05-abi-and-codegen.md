# 05. Calling Convention, Stack Discipline, And Runtime Helpers

## Call Interface Overview

This specification defines a word-based callable interface suitable for code generated by any front-end targeting TC32.

The callable interface is independent of source language syntax. It applies to:

- direct calls
- indirect calls
- helper-library calls
- startup-to-program entry transfer
- interrupt wrapper calls into ordinary handlers

## Scalar Argument Passing

Word-sized scalar arguments are assigned left to right as follows:

- first word in `r0`
- second word in `r1`
- third word in `r2`
- fourth word in `r3`
- remaining words on the stack

`safe to generate`: treat any scalar up to 32 bits as one argument word.

## Multiword Scalar Values

Multiword scalar values use consecutive word slots.

Rules:

- a 64-bit integer value occupies two consecutive words
- if enough argument registers remain, it may occupy consecutive registers such as `r0:r1` or `r2:r3`
- otherwise it spills to consecutive stack words

Return rules:

- a 32-bit result returns in `r0`
- a 64-bit result returns in `r0:r1`

## Aggregate Passing

The machine contract for aggregates is:

- aggregates of size 1 to 4 bytes may be passed in a single argument word
- aggregates of size 5 to 8 bytes may be passed in two consecutive argument words
- larger aggregates are passed indirectly by address

Return rules:

- aggregates up to 4 bytes may return in `r0`
- aggregates up to 8 bytes may return in `r0:r1`
- larger aggregates return indirectly through a hidden result pointer supplied by the caller

`toolchain obligation`: if an implementation chooses a richer language-specific ABI on top of this contract, the machine-visible lowering must still obey these register and stack shapes at binary interfaces.

## Function Pointer Representation

- a function pointer is one 32-bit word
- its stored value is `entry_address + 1`
- indirect calls and indirect jumps shall preserve this representation

## Callee-Saved State

The canonical callee-saved set is:

- `r4`
- `r5`
- `r6`
- `r7`
- `lr` when the function performs a nested call

Caller-saved registers:

- `r0`
- `r1`
- `r2`
- `r3`
- `r12`

`r7` may additionally serve as a frame pointer when a frame pointer is required.

## Indirect Call Lowering

An indirect call must preserve:

- argument registers `r0` through `r3`
- the normal return-link register behavior
- the stored `code_address + 1` function-pointer representation

Preferred lowering:

```asm
tmov r12, r0            ; or from any register holding the function pointer
tjl  __tc32_indirect_call_r12
```

Canonical helper:

```asm
__tc32_indirect_call_r12:
    tjex r12
```

Rules:

- `safe to generate`: use `r12` as the dedicated function-pointer register for an indirect-call veneer
- `safe to generate`: place the helper in ordinary executable code and call it with `tjl`
- `must not generate`: ad hoc return-address synthesis using raw `pc` arithmetic when a normal `tjl`-based veneer can be used

## Stack Layout Rules

### Required Alignment

- `sp` shall be 4-byte aligned at every call boundary.
- local stack objects requiring at most word alignment shall be rounded up to 4 bytes.

### Outgoing Arguments

- `must not generate`: reserve the entire maximum outgoing call frame once in the prologue.
- `safe to generate`: grow and shrink the outgoing call area at each call site.

### Frame Contents

A typical frame may contain:

- spilled callee-saved registers
- local stack objects
- a register-save area for variadic access
- temporary outgoing arguments beyond the register bank

## Variadic Call Support

If a front-end supports variadic calls, the callee shall make the incoming argument-register bank addressable.

Canonical rule:

- reserve a 16-byte register-save area for `r0` through `r3`
- spill `r0` through `r3` into that area before first variadic access if the function needs a variadic cursor

This makes the full incoming argument list appear as:

- register-save area first
- overflow stack arguments after it

## Prologue Rules

Safe prologue patterns include:

```asm
tpush {r4, r5, r6, r7, lr}
tsub  sp, #frame_size
```

or, when no callee-saved registers are needed:

```asm
tsub sp, #frame_size
```

## Epilogue Rules

Safe epilogue patterns include:

```asm
tadd sp, #frame_size
tpop {r4, r5, r6, r7, pc}
```

and:

```asm
load saved_return_address into top stack slot
tadd sp, #frame_size
tpop {pc}
```

Unsafe epilogues:

- any return that destroys the final result before restoring `pc`
- any multi-pop return that combines `r3` and `pc`
- any split return through a scratch register after further stack motion

## Return-Value Preservation

`toolchain obligation`: the final result register assignment must be complete before the first instruction of the epilogue sequence that could clobber or hide the result.

Example of a safe shape:

```asm
tmov r0, #0
tadd sp, #frame_size
tpop {pc}
```

## Pointer Arithmetic Rule

Short immediate add/sub forms are not safe for full-width pointer arithmetic.

Classification:

- `architecturally defined`: short immediate `tadd` and `tsub` forms
- `must not generate`: use of short immediate add/sub for pointer increment or decrement
- `safe to generate`: materialize the increment in a register and use register-form arithmetic

Safe example:

```asm
tmov r1, #1
tadd r0, r1
```

Unsafe example:

```asm
tadd r0, #1
```

## Required Runtime Helpers

The TC32 ISA does not directly define every operation a higher-level language may require.

A complete toolchain shall provide, either inline or through helper calls:

- block copy
- block fill
- block move
- integer division and remainder when admitted by the front-end
- wide integer arithmetic when admitted by the front-end
- floating-point operations when admitted by the front-end
- aggregate copy for large pass-by-value cases

The exact helper names are implementation-defined. The binary interface for calling them is not: helper calls use the same TC32 calling convention defined in this chapter.

## Atomic Operations Policy

General atomic operations are outside the TC32 machine contract in this specification.

Rules:

- `must not generate`: compare-and-swap
- `must not generate`: exclusive-access instruction sequences
- `must not generate`: multithreaded atomic library lowering
- `safe to generate`: plain load and plain store operations in a single-thread execution model
